# Trendzo Substrate Framework

## The Primitive

Your technical substrate is **the prediction**. Every feature you build is a new surface the prediction shows up through — not a separate thing. You're not building a feature list. You're building new expressions of a single prediction primitive, the same way a conversation primitive spawns artifacts, tools, memory, and skills.

Your brand primitive is **status**. The prediction is the mechanism. Status is the motivation. Every user-facing surface answers one question: *does this make the creator feel like they have an unfair competitive advantage?* If it doesn't, it's infrastructure or it's wrong.

---

## The Compounding Test

> "Does this make everything I've already built more valuable?"

If yes, build it. If it only makes itself valuable, think twice.

---

## Feature Selection Filters (run in order, stop at first No)

1. **Prediction strength** — Does it strengthen, extend, or deepen the prediction primitive?
2. **Narrative power** — Does it change how people describe Trendzo to someone who's never seen it?
3. **Retention type** — Is it a wow feature (gets attention) or a come-back-Tuesday feature (builds habit)?
4. **Moat contribution** — Does it use proprietary data a competitor can't easily replicate?

---

## Pre-Launch Build Order

1. **One deep core experience** proving the prediction works. This is your DPS/VPS score delivered through a single surface so impressive it sells itself.
2. **Invisible infrastructure** — Event Spine, prediction object lifecycle, feature flags, render contracts. This makes the next five features fast.
3. **Two to three extended surfaces** proving it's a platform, not a trick. The Pulse, Studio, Pattern Extraction — pick based on which ones pass all four filters.
4. **Onboarding flow** that makes a new user understand all of the above in 60 seconds. This ships last because it narrates what exists.

---

## Surface Extension vs. Deepening

**Extension** = a new way to interact with the prediction. New context, new audience, new modality. Example: the prediction currently lives in the admin dashboard — shipping it inside The Pulse extends it to creators.

**Deepening** = making an existing surface dramatically better. Same context, higher fidelity. Example: adding Pattern Extraction explanations to an existing prediction card.

Pre-launch you need **one deep surface + two or three extended surfaces**. Don't spread thin across six shallow extensions.

---

## Capability Prerequisites

Organize your roadmap by asking: *"If I build X, then Y and Z become trivial."*

The Event Spine is the clearest example — once `platform_events` exists with `emitEvent()`, analytics, activity feeds, the Feedback Loop, and trigger-based notifications all become listener attachments instead of standalone features.

Before starting any feature, ask what it *unlocks*, not just what it *does*.

---

## When You're Stuck

Three diagnostic questions:

1. **"What is the prediction not showing up through yet?"** — List every surface the prediction currently appears on. The gap is your next feature.
2. **"What invisible thing would make three visible things easy?"** — You're probably stuck because the next user-facing feature requires infrastructure you haven't built. Build the infrastructure.
3. **"If I shipped nothing else and launched tomorrow, what's the one thing that would make someone say 'holy shit' vs. 'that's cool'?"** — Go deepen that thing.
